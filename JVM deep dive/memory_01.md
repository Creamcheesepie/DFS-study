# 자바 메모리 영역과 메모리 오버플로우
## 들어가며
C, C++r과 달리 자바에서는 가상 머신이 제공하는 자동 메모리 관리 메커니즘 덕에 메모리 할당과 해제를 굳이 열심히 해줄 필요가 없어졌다.
다만, 이런 방식으로 인해서 메모리 관련 문제가 발생할 경우, 가상 머신의 메모리 관리 방식을 이해해야 문제를 해결할 수 있을 때가 많아졌다.

## 런타임 데이터 영역
자바 가상 머신은 자바 프로그램을 실행할 때 필요한 메모리르 몇 개의 데이터 영역으로 나누어 관리한다.
이 영역들은 각각 어떤 목적을 가지고 만들어진다. 또 생성, 삭제 시점을 따로 가지기도 한다.
어떤 영역은 가상 머신 프로세스의 시작과 동시에 만들어지며, 어떤 영역은 사용자 스레드의 시작, 종료에 맞춰 생성되고 삭제된다.

### 프로그램 카운터
프로그램 카운터 레지스터는 작은 메모리 영역으로, 현재 실행 중인 스레드의 바이트코드 줄 번호를 표시한다고 생각하면 쉽다.
자바 가상 머신의 개념 모형에서는 이 카운터의 값을 바꾸어 다음에 실행할 바이트 코드 명령어를 선택하는 방식으로 동작한다.
프로그램의 제어, 분기, 순환, 점프 등을 표현하며, 예외 처리나 스레드 복원등의 모든 기본 기능이 이 표시기를 통해 이루어진다.
자바 가상 머신의 멀티스레딩은 CPU 코어를 여러 스레드가 교대로 이용하는 식으로 구현되어, 특정 시간에 각 코어는 하나의 스레드의 명령어만 수행할 수 있다.
따라서 스레드 전환 후 이전에 실행하다 멈춘 지점을 정확하게 복원하기 위해서는 스레드 각각에 고유한 프로그램 카운터가 필요하다.
이를 위해 각 스레드의 카운터는 서로 영향을 주지 않는 독립된 영역에 저장되고, 이 메모리 영역을 스레드 프라이빗 메모리라고 한다.
스레드가 자바 메서드를 실행중일 때에는 실행 중인 바이트 코드 명령어의 주소가 프로그램 카운터에 기록된다.
반면, 스레드가 네이티브 메서드(Java 언어 외부에서 구현된 메소드를 Java코드에서 호출하는 메소드)를 실행 중일 때에는 프로그램 카운터 값은 Undefined가 된다.
프로그램 카운터 메모리 영역은 자바 가상 머신 명세에서 OutOfMemoryError 조건이 명시되지 않은 유일한 메모리 영역이기도 하다.

### 자바 가상 머신 스택
자바 가상 머신 스택도 '스레드 프라이빗'하며, 연결된 스레드와 생성 삭제 시기가 일치한다.
가상 머신 스택은 자바 메서드를 실행하는 스레드의 메모리 모델을 설명해준다.
각 메서드가 호출될 때마다 자바 가상 머신은 스택 프레임을 만들어 지역 변수 테이블, 피연산자 스택, 동적 링크, 메서드 반환값 등의 정보를 저장한다.
그런 다음 스택 프레임을 가상 머신 스택에 푸시하고, 메서드가 끝나면 팝 하는 일을 반복한다. 
보통 스택이라고 하면 자바 가상 머신 스택에서 지역 변수 테이블을 가르킬 때가 많다.

#### 지역 변수 테이블
지역 변수 테이블에서는 자바 가상 머신이 컴파일때 알 수 있는 다양한 기본 데이터 타입, 객체 참조, 반환 주소 타입(바이트 코드 명령어의 주소)를 저장한다.
지역 변수 테이블에서 이 데이터 타입들을 저장하는 공간을 지역변수 슬롯이라고 한다. 슬롯 하나의 보통 크기는 32비트고 64비트에 해당하는 double 등의 타입에서는 슬롯 두개를 차지한다.
지역 변수 테이블을 구성하느 데 필요한 데이터 공간은 컴파일 과정에서 할당된다. 자바 메서드는 스택 프레임에서 지역 변수용으로 할당할 공간의 크기가 완벽하게 결정되어 있고, 메서드 실행 중에는 절대로 바뀌지 않는다.
여기서 '크기'는 변수의 슬롯 개수에 주의해햐 하는데, 가상 머신이 변수 슬롯을 구현하는데 사용하는 메모리의 '실제 크기'는 가상 머신을 어떻게 구현하는가에 따라 완전히 달라질 수 있다.
(슬롯 하나의 크기가 32비트, 64비트 또는 그 이상잏 수도 있다. 자바 가상 머신 명세에서는 스택 메모리 영역에서 StackOverFlowError와 OutOfMemoryError 두가지 오류가 발생할 수 있게 정의되어 있다.
StackOverFlow의 경우에는 스레드가 요청한 스택 깊이가 가상 머신에서 허용하는 깊이보다 클 때 발생한다. OutOfMemoryError는 스택을 확장하려는 시점에 여유 메모리가 충분치 않을 때 발생한다.

### 네이티브 메서드 스택
이 스택은 가상 머신 스택과 매우 비슷한 역할을 하는데, 가상 머신 스택은 자바 메서드(바이트 코드)를 실행할 때 쓰이고, 네이티브 스택은 네이티브 메서드 실행 시 쓰인다는 정미다.
이 스택에서 메서드를 어떻게 표현할지에 대해 자바 가상 명세는 따로 명시해두지 않았다. 가상 머신 구현자가 필요한대로 자유롭게 구현이 가능하며, 필요하다면 네이티브 메서드 스택과 가상 머신 스택을 하나로 합칠 수도 있다.
가상 머신 스택과 같이 StackOverFlowError와 OutOfMemoryError를 일으킬 수 있다.

### 자바 힙
자바 힙은 자바 애플리케이션에서 사용할 수 있는 가장 큰 메모리에 해당한다. 모든 스레드가 공유하며, 가상 머신이 구동되는 시점에 만들어진다. 자바 힙은 객체 인스턴스를 저장하며, 거의 모든 객체 인스턴스가 여기 할당된다.
'거의'모든 객체 인스턴스인 이유는, 향후에는 필요에 따라 값 타입도 지원할 것으로 보이기 때문이다. JIT 컴파일 기술의 발전에 따라 현시점에서도 스택 할당과 스칼라 치환 최적화 방식이 살짝 달라졌다.
이 때문에 모든 자바 객체 인스턴스가 힙에 할당된다는 설명이 절대적으로 옳다고 하기에는 애매해지고 있다.

자바 힙은 가비지 컬렉터가 관리하는 메모리 영역이다. 어떤 책에서는 GC 힙으로 쓰이기도 한다. 메모리 회수 관점에서 대다수 현대적인 가비지 컬렉터는 세대별 컬렉션 이론을 기초로 설계되어 있다.
따라서 자바 힙을 설명할 때 '신세대', '구세대', '영구 세대', '에덴 공간', '생존자 공간에서(from the survival space)', '생존자 공간으로(to survivor space)'과 같은 용어가 자주 등장한다.
이러한 용어는 후술하는 장에서 자주 볼 수 있을 것이다. 다만 이 영역 구분은 가비지 컬렉터들의 일반적인 특성이나 설계 방식에 불과할 뿐, 반드시 이 형태로 메모리를 구성해야하는 것을 아니라는 점을 양지해야 한다.
자바 가상 머신 명세의 자바 힙 절에서는 세부 영역 구분에 대한 묘사가 없다. 

다른 여러 책들에서는 "자바 가상 머신의 힙 메모리는 신세대, 구세대, 영구 세대, 에덴, 생존자 공간 등으로 나뉜다"고 설명한다. 2010년에 G1 컬렉터가 등장한 전후로 핫 스팟 가상 머신은 주류가 되었다.
핫스팟의 가비지 컬렉터들은 모두 신세대 컬렉터와 구세대 컬렉터를 조합해 동작하는 전통적인 세대 구분을 따르는데, 이에 따르면 앞의 설명이 크게 틀린게 없다.
하지만 현 시점의 가비지 컬렉터 기술은 세대 단위 설계를 따르지 않는 컬렉터들이 나와 앞의 설명에 대해 따져 볼 것이 많아졌다.
